Element

This column contains the local name of each XML node in resultXML. Indentation signifies hierarchy in the XML structure.

Cardinality

The value in this column indicates how many of the given nodes are expected in resultXML. One (1) indicates that the node is required. One-to-many (1..*) indicates that at least one occurrence of the node is required. Zero-to-one (0..1) and zero-to-many (0..*) indicate optional nodes. Required nodes can be children of optional nodes; in this case, the child is required only if the parent is present.

Remarks

When the status is returned as "success" and with an actionType of "conflict," the collision handler has successfully resolved the data collision. In this case, the collision handler is instructing the calling client to try to save the winning record again (the winning record is usually the offline record), using an updated onyxTimestamp value. The user is not sent a conflict message.

If the status is returned as "failure," then the <actionType>, <primaryXml>, <secondaryXml>, and <baselineXml> nodes do not contain valid data and should be ignored by the calling client.

If the status is returned as "success":

  • If the actionType is "conflict": The <primaryXml> node contains the onlineXml data that represents the record that "won," the <secondaryXml> node contains the offlineXml data that "lost," and the <baselineXml> node contains the data against which the online and offline data were compared.
  • If the actionType is "retry": The calling client is instructed to process the record again, using the data contained in the <primaryXml> node. Normally, the <primaryXml> node contains the offlineXml data with an updated onyxTimestamp, so the retry of the OEAS operation will succeed. The <secondaryXml> contains the onlineXml data; this data is provided for a potential future enhancement that may allow the calling client to inform the system/user of what was overwritten.
  • If the actionType is "noOp": The <primaryXml>, <secondaryXml>, and <baselineXml> nodes will not be present.

View the output resultXml

How to read the output XML:

  1. First look at the <status> node, which tells us whether or not the collision has been resolved. In this case, the status is "success," which means that the collision handler succeeded in resolving the conflict using the algorithm indicated in the input XML.
  2. Next, take a look at the <actionType> node. The actionType node can contain one of three values: Retry, No-Op, and Conflict. In this case, the actionType is Retry, meaning that the synchronization service will again try to save the record to the OEDB, this time by saving the record that it has determined as the "winner" of the data conflict.
  3. Finally, review the contents of the <primaryXml> node, which contains the "winning" record that the synchronization service will now try to save to the OEDB. You can ignore the <secondaryXml> and <baselineXml> nodes for now, because they will be ignored by the synchronization service.
