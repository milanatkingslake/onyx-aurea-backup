Function mapping use cases
Use cases

The following use cases illustrate reasons to use function mapping:

  • Case 1: Type conversion

Sometimes you have a minor data type mismatch between display data and how it is stored. This is the case with OEAS GUID values. These are displayed in the user interface formatted with dashes, a 36-character string. In the database these are stored as 16 byte hex numbers to minimize space requirements.

  • Case 2: Automatic ID generation

Some primary key values are generated in the database. You use a function call to set these values.

  • Case 3: Functional denormalization

Sometimes you use properties that must be "functionally de-normalized." For example, you want to display both an owner ID and an owner name on an object but store only the owner ID. In this case, you retrieve the owner name property from another table using a function. You use a function like this when it is not possible to use a denormalized property because of varying owner types.

  • Case 4: Line numbers on collections

For example, you want to use a database function to manage the sequencing of IDs on detail records. OEAS maps the lineId of the forecastLines object to a complex function: dbo.ofForecastDetailGetNextLineNumber.
