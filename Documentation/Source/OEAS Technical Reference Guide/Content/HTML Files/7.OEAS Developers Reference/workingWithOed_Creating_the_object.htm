Creating objects

You must make two important decisions before creating a new business object.

  1. Is the object type domain or transactional?

For more information on making this decision, see the Object categories topic. This decision is important because it effects how SQL generation treats the object. It determines what maintenance properties the object is expected to possess and what types of relationships it may have to other objects. It has a similar impact on the design of the database table used to persist object properties.

  1. Will the object need to persist a property that can contain a very large amount of text?

Creating an object

Based on your object creation scenario, you should start with the appropriate base set of object properties and table columns. Then add the other object properties your implementation requires. If possible, design the corresponding database table in such a way that the properties and columns are similar. You can simplify your work in later stages by designing with one-to-one correspondence and similar data types up front. Once you define the properties and columns, you still have a fair amount of work to do. You must specify the following:

  • Primary key properties and primary key columns.
  • How the object is mapped to the table.
  • What relationships exist between your new object and other objects in the model.
  • What methods your object requires.
