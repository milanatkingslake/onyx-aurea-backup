Customizing the Business Objects

You can modify the default behavior of the business objects so that they function according to the needs of your company. Before you modify them, it is important to understand the relationship between the compiled COM step components and the logical model in the Onyx Enterprise Dictionary (OED).

The OED defines each business object and the properties, methods, and steps associated with that object. The Onyx Transaction Manager (OTM) uses these definitions, at run-time, to implement business processes and data access transactions. When a client invokes a method, for instance, the OTM queries the OED cache for the steps associated with that method, and then executes those steps in the specified order. If the step is a "com" step, it instructs the OTM to instantiate a step component.

For example, imagine that a client invokes the execute method on the foo object. The foo.execute method consists of a single step that calls a step component responsible for managing foo data in the Onyx Enterprise Database. Suppose, however, that you want the execute method to perform two separate actions, not just one. Using the Object Designer, you can customize the execute method so that it consists of a second step, such as:

  • Step 1: Call the CustomStep.Foo step component to manage a foo record in the database.
  • Step 2 (new step): Call another step component to automatically send literature to the customer.

For any given method, you can add as many steps as you need.

Depending on your customization of the business objects in the OED, you might or might not have to create your own step component. OEAS provides many opportunities for customization.

Add a custom object

Create a corresponding database table. Using Object Designer, describe the table in the physical model, add an object to the logical model, and create a mapping between the logical and physical models.

OEAS automatically implements CRUD methods for custom objects. If the custom object requires special logic other than create, retrieve, update, or delete, you have to implement the logic in either a custom step component, custom database logic, or both.

Disable a default OEAS object

Resolve cross references.

 

When you disable an object, the Object Designer verifies that doing so does not create incompatibility issues or otherwise break the system. When the Object Designer discovers such a conflict, it notifies you of the cross-dependencies for that object. In order for the system to function correctly, you must resolve all cross-dependency issues.

Add a property

Add a column to an existing database table. Using Object Designer, describe the column in the physical model, add a property to the logical model, and map the logical property to the physical model.
