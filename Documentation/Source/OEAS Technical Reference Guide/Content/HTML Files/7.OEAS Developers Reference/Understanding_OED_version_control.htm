Understanding OED version control

Employing standard version control on the OED can save you a lot of time and effort if, for any reason, you need to reimplement a former version of the OED. Therefore, when you customize the OED or any portion of a definition, be careful to maintain the version numbers. Before you begin a customization, save duplicate copy of the OED production copy (OnyxEnterpriseDictionary.xml) to a safe location. Also label this file so that if you ever need to return to a former version of the OED, you will know both the version and the general contents of the OED.

If your customizations cause the OED to function improperly, you can always reinstall it from the OEAS installation disk, but you will, of course, lose any customizations that you made in the meantime.

Version numbers

Each business object, object physical map, relationship, and table in the Object Designer is assigned at least one version number: the version of the item as shipped by Onyx. When you customize these entities or create custom ones, you can record the version number for the item using any scheme that makes sense to you and your organization.

The full version number for an element concatenates the default Onyx version number and the custom version number. For example, if you customize an entity, the version XML element for that entity looks like this:

<version>7.06.00.000.1.00.001</version>

where:

6.00.000 is the default Onyx version number

1.00.001 is your custom version number

Important: The version number is maintained only for the highest level entities. Therefore, when you customize any portion of a definition, you should update the custom version number for the affected entity.
