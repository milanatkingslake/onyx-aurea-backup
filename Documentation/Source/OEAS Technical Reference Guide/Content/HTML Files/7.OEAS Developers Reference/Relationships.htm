Relationships

The relationships section of the OED contains the metadata that defines how business objects are related to other business objects. The following table summarizes the relationships metadata.

Metadata Description

Relationship

A connection or correlation between two objects

Primary object

The object whose primary key defines the relationship

Secondary object

The object that contains a foreign key to the primary object

Key maps

Corresponding properties of the related objects; the properties involved will always be a primary key on the primary object, and a foreign key on the secondary object, and an object type on the secondary object for polymorphic relationships.

The same two objects can be related using more than one relationship when appropriate. For example, an individual can own an incident, but an individual can also be the primary contact for an incident. This situation requires the definition of two relationships between the individual and incident objects. Note that individual is the primary object in both relationships.

Relationships are used for the following purposes:

  • Implementing cascade deletes on transactional objects
  • Enforcing referential integrity between transactional objects and for polymorphic relationships
  • Determining how to retrieve property values from one object to another (denormalization)
  • Enforcing concurrency control between tightly-coupled objects.

For instance, the incident object includes a relationship defined on its ownerId property and the primaryId properties of company and individual objects. These relationships are what define that the valid ownerType values for incident are 1 and 2 (individual and company), and that when an incident is created, the ownerId should be validated against existing active data in the appropriate tables (individual and company).

Relationship definitions appear in Object Designer under the business Object node for the secondary object.
