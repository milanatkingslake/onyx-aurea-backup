Uniqueness Constraint attributes

When you select Object Keys in the Object Designer, the Uniqueness Constraint attribute displays in the edit area.

The attribute is defined as follows:

Attribute Name

Description

Uniqueness Constraint

Select the key you want to define as the uniqueness constraint. This list includes objects keys marked for deletion and objects keys that are disabled, both of which are marked accordingly. Invalid object keys are not listed.

Note: Select an alternate key from the drop down list. Do not select the primary key for the object. Generating code to enforce a uniqueness constraint on a primary key is unnecessary and will degrade performance. Onyx recommends only using the uniqueness constraint on domain objects.

Adding new constraints

When adding new constraints to an existing object or a custom object, a database administrator must update the schema to grant SELECT privilege to OnyxPublic role on any physical table related to the object key. The SQL generation process generates run-time SQL that checks the uniqueness constraint against one or more database tables depending on the composition of the object key. OEAS makes database calls as a member of the OnyxPublic role which must have SELECT privilege on the tables involved in the constraint.

For example, suppose a customer has a foo object which maps to the foo table in OEDB. The customer adds a new uniqueness constraint on fooPrimary, an object key named after the object and property composing the key. To prevent run-time errors, an OEDB administrator updates the database schema by running the following query directly on the OEDB using a SQL management tool.

GRANT SELECT ON foo TO OnyxPublic
