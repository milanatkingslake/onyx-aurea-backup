The following is a grid representation of the structure of the input parameter called compareXML.

View the grid legend

Element

This column contains the local name of each XML node in compareXML. Indentation signifies hierarchy in the XML structure.

Cardinality

The value in this column indicates how many of the given nodes are expected in compareXML. One (1) indicates that the node is required. One-to-many (1..*) indicates that at least one occurrence of the node is required. Zero-to-one (0..1) and zero-to-many (0..*) indicate optional nodes. Required nodes can be children of optional nodes; in this case, the child is required only if the parent is present.

Comments

This column contains notable information about the node.

  • remove
  • noOp

1

A container for design-time data that is passed to the collision handler.

The content of this node is stored as a string in the SubscribableObject table (and in its corresponding cache, when put into memory).

A container for the configurable data.

A container for the configurable data that instructs the collision handler how to resolve the data collision.

An enumeration that specifies which conflict resolution algorithm to use when attempting to resolve the data collision. Valid values are:

  • lastUpdateWins
  • onlineWins
  • customer

1

Indicates whether the OPS synchronization service should attempt to persist the object again.

This is a Boolean value, which is set to True (1) only for incident and task objects.

A container of the object properties that the collision handler is to ignore when attempting to resolve the data collision.

0..*

A property that the collision handler is to ignore when attempting to resolve the data collision.

Remarks

If you have customized the collision handler, the <config> node and its child nodes may not be required.

The <offlineAction>, <onlineSubscribeAction>, and <offlineSubscribeAction> nodes are passed in by the OEP-Offline client, along with the <offlineXml> and <baselineXml> nodes.

Example

View the input compareXML

How to read the input XML:

  1. First review the first seven nodes of the input compareXML, which contain contextual information regarding the records that are being sent to the collision handler. From this context XML, you can see that FredS is the user who is attempting to synchronize an updated incident record.
  2. Next, expand the <extraData> node and review the algorithm that will be used to determine how the collision will be handled. In this case, the algorithm is "lastUpdateWins," which means that whoever last updated the record will succeed in saving their updates. The <readOnlyProperties> node contains a list of properties that the collision handler will ignore when comparing the conflicted records, meaning that they will not influence how the collision handler may resolve the conflict.
  3. Finally, compare the three copies of the incident record that is being passed to the collision handler. The <offlineXml> node contains the version of the record that FredS is attempting to upsync from OEP-Offline to the OEDB. The <onlineXml> node contains the version of the record that is currently in the OEDB. The <baselineXml> node contains the version of the record that FredS had last downsynced from the OEDB, and reflects the original state of the record before he had updated it. The baseline XML is part of the official offline record, and is always passed along with the offline XML.
