retrieveList method

The following table describes the behavior to expect for retrieveList methods.

Behavior based on the stored procedure return code value defines the entire result set. In this case, the return code column is listed as not applicable (N/A) because it is not included in the result set. For stored procedure return code values less than or equal to zero (<0), the result set must contain a column for each columnDefinition defined in the rowSet parameter of the given object and method. The columns in the result set must be returned in the order they are defined in the columnDefinitions. The middle tier uses the column names and/or aliases to populate the columnName of each columnDefinition. For more information, see the retrieveList method.

When the contentOut on a retrieveList method's rowSet parameter is "dynamic" (as it is for the searchQuery.execute method, for instance), the procedure has total control of the result set and it can be any number of columns, even changing from one call to the next.

Behavior only applies to the single row when the return code column is applicable.

The procedure is successful and the result set parameter contains the rows of data being successfully returned from the database.

The return code parameter value is converted to a positive value and the MOD result of the value corresponds to the national language beginning grid heading value used to generate the column headings for the columns being returned.

In most instances when a database result set is returned, the column heading values are controlled by the OED. However, in cases where the stored procedure contains some logic that would necessitate one or more possible different sets of column headings, you can control the column headings used by passing a parameter value of -300,000,000 or less.
