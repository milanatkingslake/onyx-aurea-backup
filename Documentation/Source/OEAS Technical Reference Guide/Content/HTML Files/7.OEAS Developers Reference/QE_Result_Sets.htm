Result Sets

Result sets

The Query Engine returns a result set in a two-dimensional SafeArray (equivalent to a standard Variant array in Visual Basic). Each row in the result set is duplicated in a separate row in the returned array.

The first dimension in the array represents the rows in the result set. The second dimension represents the columns. To get the record count of a result set, check the upper bound of the first dimension in the array. To get a column count, check the upper bound of the second dimension in the array. The lower bound of both dimensions is always zero, so the count is the upper bound + 1. If the command returns no records, the Variant is empty.

Data types

Each element in the array is of type Variant. Custom step components must have some familiarity with the data they retrieve from the application database to make the proper data conversions and validations. Not all data types behave as you might expect. The following sections provide important information about some limitations and caveats regarding result set data manipulation.

Variable Subtypes

The subtype of four SQL data types, float, real, decimal, and numeric, varies depending on the retrieval method (ad hoc SQL vs. stored procedures). When retrieved through a stored procedure, the subtype for an element of these types is VT_BSTR (Visual Basic String), but when retrieved using ad hoc SQL, the subtype is VT_R8 (Visual Basic Double). Step components can check the subtype to make a legal conversion.

For float and onyxfloat data types: OEAS returns many data types as strings, regardless of the actual data type that is stored in the database. Data stored in the database with a data type of "float" or "onyxfloat," when returned as a string, sometimes becomes less accurate because of the way in which the number is rounded during the conversion. For example, if the number 10.1234 is stored in the database as a float or onyxfloat data type, a retrieve method may return the number as a string with the value 10.12399999999. To ensure that the number is accurately returned to the calling client, you must first convert the string data type to the appropriate data type that the calling client expects.

Unsupported data types

To maximize OEAS performance, the Query Engine was designed to support the transfer of most, but not all, of the data types provided by a relational database. The unsupported data types are not required for use with a standard installation of the OEAS. A column that contains an unsupported data type cannot be accessed successfully using the Query Engine. If you have a custom table with a column like this, build a step component that can communicate with the database using a different data-access technology (for example, ActiveX Data Objects (ADO) or OLE DB). Alternatively, you can write a stored procedure or database function that returns a supported data type for a column with an unsupported data type. Depending on logical method type, a procedure or function can be referenced as a replace procedure or a pre- or post- procedure on the method call without having to write a step component.
