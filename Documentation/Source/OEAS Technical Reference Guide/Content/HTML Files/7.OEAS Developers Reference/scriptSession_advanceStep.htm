advanceStep (scriptSession)

advanceStep method

The advanceStep method of the scriptSession object persists the transactional data for the current step and returns the domain data for the next step.

Remarks

You can call this method when you want to persist transactional data for the current step and move to another step in the script. If both optional parameters are blank, the engine performs in order the following functions:

  1. Updates variables from responses
  2. Persists the transactional data from the current step
  3. Processes post method calls for the current step
  4. Evaluates the branching rules for the current step
  5. Deletes local variables for the current step
  6. Persists the script session transactional data
  7. Calls the loadStep method

If you specify either nextSessionStepId or nextStepId, the advanceStep method does not evaluate any branching rules and instead returns the specified step. For nextSessionStepId, it returns both transactional and domain data. For nextStepId, it returns only the domain data. If you specify both, it returns both transactional and domain data.

The advanceStep method supports the ability to assign scriptSessions to a new owner. The engine respects any changes to context variables _ownerid and _ownertype made by script step method calls. For example, you can do this with an individual insert step method call. The insert method returns the primary key of the individual in an output detail. You can assign this output detail to the _ownerId context variable. If you need to change the ownerType, you can do so using the customData node. In the method call design, you create both input and output details using /customData/ownerType as the element path. In the script design, you bind the input detail to a literal or variable value and the output detail to _ownerType context variable. Prior to updating the scriptSession object in the database, the script engine modifies the ownerId and ownerType children of the scriptSession element from the _ownerid and _ownertype context variables in the run time XML. The engine copies the values from the context variables and overwrites the values in the scriptSession run time XML.

Example

This example uses a script which does the following:

  1. Uses a script that prompts the user to search the database for the ID of a caller.
  2. Asks if the user wants to continue or end the script.
  3. If the user chooses to continue, the script asks them if they want to enter an incident for the caller. The user can choose to enter an incident or end the script.
  4. If the user chooses to enter an incident, the script displays the new incident ID.
  5. In the final step, displays the caller's ID and the new incident ID.

The following example XML, based on this script, shows an advanceStep call to move from the "identifyCaller" step (1) to the next step based on branching rules. In order to have the engine evaluate branching rules, the caller only passes in the scriptSessionId and scriptSessionXml. The run-time XML contains script session, session step, and session prompt response data as well as the context variables. The responses contain the ID and name of the caller and the user's choice to continue with the script.
