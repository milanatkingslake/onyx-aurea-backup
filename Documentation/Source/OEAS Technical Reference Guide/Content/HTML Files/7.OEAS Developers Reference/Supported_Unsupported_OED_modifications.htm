Supported/Unsupported OED modifications

Supported/Unsupported OED modifications

Prior to making modifications, determine which business objects are used by the Onyx Enterprise Studio tools and any other user interfaces so that you know what is affected by the modifications.

Supported Modifications

When you use the Object Designer to create custom business objects, Onyx expects you to complete all necessary attributes for your custom objects and specify its presence in the physical model and object physical maps.

Because OEAS is predominately data-driven, it is imperative that the OED is accurate. OEAS uses the logical model, physical model, and the mappings of the OED for most of its operations. An incorrect OED leads to unpredictable behavior and method failures.

Important: Before you publish your modifications in your OEAS production environment, carefully test them in an OEAS test environment. Changes that you make to the OED can affect the functionality of dependent products such as the Onyx Enterprise Studio and the Onyx Employee Portal.

Non-Data-Driven Attributes

Onyx supports the modification of the following attributes in the OED, since these attributes are not data-driven in this version of OEAS:

  • Modifying the description of an item. The Description field is used primarily for documentation and maintenance purposes. OEAS does not rely on the value of this attribute.
  • Identifying whether the item has been released. The Released field is used primarily for documentation and maintenance purposes. OEAS does not rely on the value of this attribute.
  • Identifying who installed the item. The InstalledBy field is used primarily for documentation and maintenance purposes. OEAS does not rely on the value of this attribute. However, Onyx may use this field during upgrades of OEAS.
  • Identifying a custom version number for a modified object. The Custom Version field is used primarily for documentation and maintenance purposes. OEAS does not rely on the value of this attribute.

Data-Driven Attributes

Onyx supports the modification of the following attributes in the OED. However, because these attributes drive the behavior of platform components, you must understand the impact of modifying them. For more information on attributes, see OED Concepts.

Step Type, Step Data, Additional Data, Procedure Maps, and Sort Property attributes

The values of the step type and step data are passed to the step components as a parameter of the Execute method of the IStep interface, which is the primary client call into the OEAS middle tier. If these values are not correct, the execution of the client call fails. The additionalData attribute is also passed to the step component identified by ProgID identified in the step data attribute. Some of this information is intended to be redundant: the values for individual component items (stepName, stepMethod, stepObject, and so on) should be in agreement with the corresponding OED elements. For more information, see the Methods section of OED Concepts.

Property attributes

Onyx supports changing the default value of property constraints, changing the default primary key and how it is generated, and defining properties as denormalized. A denormalized property definition identifies a relationship that enables the retrieval of the denormalized property value. For more information, see the Properties section of OED Concepts.

Object Relationship attributes

The OED defines object relationships, identifying the primary and secondary objects, the properties involved, and the relationship type. The type determines how a delete affects related objects. For more information, see Relationships in OED Concepts.

Physical Model attributes

The OED describes the data tier, both physical structures and procedures used on the data tier. You can define tables, columns, database actions, rules for row filtering, and rules for data security. For more information, see the Physical Model in OED Concepts.

Unsupported modifications

Onyx does not support the following modifications to the OED:

  • Any modifications made directly to the OED source file (OnyxEnterpriseDictionary.xml). You must use the Object Designer to customize the OED. Customizing the OED directly can produce unexpected results.
  • Any modifications made to objects that do not display in the Object Designer, such as "businessObjectDefinition." These objects are system objects integral to the OEAS architecture. Customizing the system objects can break core OEAS functionality.
  • Any modifications to the row, rowSet, and columnDefinition objects.
  • Any modifications made to the default OEAS objects, properties, methods, parameters, and steps, other than the supported modifications listed above. Whenever you customize default OEAS objects, test your customizations in a test environment before publishing them to your production environment.
  • Do not disable default OEAS business objects, since doing to will severely impact OEAS functionality. You should enable and disable only custom objects and methods.

    When you disable an item and validate your changes, the Object Designer informs you of all broken cross-dependencies resulting from the disable. Resolve all cross-dependencies before publishing your changes.
