Properties and Columns (transactional)
Properties and columns

For this example, you can assume that the company running OEAS inserts database records in the company table that describe their own firm and any subsidiary companies they may own. This enables them to track both parties in a contract.

You need a primary key. You also want to know what type of contract is being recorded. The parties involved in the contract must also be recorded. Assume that a company object owns the contract (the OEAS license holder), but that the contracting party (client) may be with a company or an individual. You also want pertinent date information. You should also have some informative summary of the contract contents in a text field. You can include other custom properties, but this is enough for now. You should also include any standard properties required by SQL generation or Onyx standards. Because you also need a table to persist our object's data, you need to know what standard database columns are required by SQL generation or suggested by Onyx standards.

In this example you name the property for the party holding the contract "contractor" and the property for the second party "contractee." The type of the contractor property is known as a company ID. The type of the contractee property cannot be predetermined; it can be the ID of either a company or an individual. To resolve this relationship at run time, you add a property to describe the object type (contracteeType) which will hold an integer value taken from an enumerated list of objects.

After careful consideration, you conclude that most of the non-maintenance properties should be marked as "required" with a property constraint. The only exception is the expirationDate. Because contracts can be written so they don't expire, this property should not be required. When entering the properties in the Object Designer Onyx standards indicate that the maintenance properties should be marked as "read only."

Enter the table structure in the physical model section of the Object Designer first. Mark all of the columns as "not null" except for any column that corresponds to a "not required" property. On this table the expiration_date column is not required, so it is "nullable."

The next step is to create the business object in the logical model section of Object Designer. Be sure to leave the check the box for "Domain Object" empty for this transactional object. Enter the properties and their "required" property constraints.
