Object Categories

Transactional and non-transactional objects

OEAS business objects can be divided into two categories: transactional and non-transactional.

Transactional objects represent the data that defines the customer records (company and individual records) in your database. Transactional objects also represent the data that defines the activities that are associated with these customer records, such as incidents and tasks. In general, transactional data is created and maintained using an end-user UI, such as Onyx Employee Portal.

Non-transactional objects, on the other hand, represent the data that defines your business environment. This data is known as domain data. These non-transactional objects and non-transactional (domain) data are used to classify and describe the customer and activity records themselves. In general, domain data is created and maintained using an administrative UI, such as Onyx Enterprise Studio.

Transactional Objects

In OEAS, transactional objects represent customer data such as company and individual records, phone numbers, incidents, literature orders, and so on. Using transactional objects, you capture important information about your company's interactions with customers.

Transactional objects can insert, update, retrieve, and delete dynamic information in the database. You can customize OEAS by changing the properties and methods of these objects, thereby modifying the business logic of OEAS.

Most transactional objects support record locking, either directly or indirectly, so that one user does not unintentionally overwrite another user's modifications to a customer record.

Non-Transactional Objects

Non-transactional objects represent domain data (data that is universal to all implementations) that applies to many customer objects and transactions. Non-transactional objects are typically referenced system-wide, and can include system parameter and user objects.

When you install OEAS, default domain data is installed in the Onyx Enterprise Database to facilitate your customer relationship management (CRM) activities. Examples of non-transactional objects include statuses for incident and task records, incident categories, and lists of products. Other examples include non-CRM-specific objects such as countries and regions, currency codes, language codes, SIC codes, and so on.

Non-transactional objects tend to be more static than transactional objects, and most of them are maintained by system administrators. These objects contain data that end-users select, usually through combo-boxes (also known as drop-down list boxes), to apply to a customer interaction.

Non-transactional objects typically do not support record locking.
