executeForEMS (actionBatch)

executeForEMS method

The executeForEMS method of the actionBatch object is called by the email monitor subsystem (EMS). For incoming email messages that are detected in monitored mailboxes, EMS constructs the actionBatchXml parameter, which provides data for associating incoming messages with customers, email threads, attachments, and incidents, as appropriate. Before executing the actionBatchXml, the executeForEMS method calls the OnyxTransformStep step component to modify and enhance the actionBatchXml from EMS so that it inserts an incident related to the message, if necessary. You can enable and modify parts of this XSLT transform so that an auto-reply email is created and sent.

Note: The OnyxTransformStep is disabled in the default configuration of this method. Using OnyxTransformStep requires that you enable the step using the Object Designer. Also, the XSLT template that Onyx supplies to be used with EMS needs to be configured for your implementation of OEAS and OEP.

Parameters

Parameter Direction Data type Content type Description

actionBatchXml

in

string

n/a

XML that defines the actions of the batch processing. This XML must be encoded.

statusXml

out

string

n/a

The status of all the calls processed by the execute method.

Remarks

The stepAdditionalData for the OnyxTransformStep specifies the path of the XSL template that is executed and which node in the stepPackage that is transformed. When you view the OnyxTransformStep of the executeForEMS method using the Object Designer, the stepAdditionalData text box contains the basic configuration for the step. The default stepAdditionalData for the specific step is shown below, and it specifies the location of XSL transform at the time of OEAS installation.

stepMethod=executeForEMS;

stepObject=actionBatch;

templatePath=C:\Program Files\ONYX\AppServer\Onyx\templates\OnyxEMSIncidentInsert.xsl;

escapedNode=stepPackage/parameters/actionBatchXml

The templatePath is either the absolute path of the template (as shown above) or a relative path to the location of the dllhost.exe file on the application server (typically, c:\windows\system32).

With minor modifications, the XSLT template supplied by Onyx for the executeForEMS method is capable of the following functionality:

  • When the incoming email message is the first message of an email thread, the template creates an incident and associates the message with incident. The properties of the incident can be generated from emailMessage object instance properties or literal values specified in the XSLT template
  • When the person who sent the email message is resolved to an individual or a company object instance, the template makes the individual or the company's primary contact the owner of the incident.
  • When the incoming email message is a part of an existing email thread that is tracked in Onyx, the template updates the incident associated with the thread by associating the new message with the incident.
  • Sends an auto-reply to the sender of the incoming email message.
  • Supports multiple monitored mailboxes, such as support@company.com and sales@company.com.

Note: In OnyxEMSIncidentInsert.xsl, do not modify the line that reads:

<xsl:if test="$dummyCustomerId!='ENTER DUMMY CUSTOMER ID HERE'">

<xsl:stylesheet xmlns:xsl="http://www.w3.org/1999/XSL/Transform" version="1.0">

<!-- Start processing at the root node of the stepPackage -->

<xsl:template match="/">

<xsl:if test="$dummyCustomerId!='ENTER DUMMY CUSTOMER ID HERE'">

    <stepPackage>

        <xsl:copy-of select="stepPackage/context"/>

        <parameters>

            <actionBatchXml>

                <root>

                    <actions>

                        <!-- Following is the whitelist of ojects/methods that are propagated in the transform. -->

                        <xsl:copy-of select="$actionsNode/action[logicalObjectName='emailMessage'][logicalMethodName='insert']"/>

                        <xsl:copy-of select="$actionsNode/action[logicalObjectName='fileLink'][logicalMethodName='saveCollection']"/>

                        <xsl:copy-of select="$actionsNode/action[logicalObjectName='emailAssociation'][logicalMethodName='saveCollection']"/>

                        <xsl:copy-of select="$actionsNode/action[logicalObjectName='emailUserAssociation'][logicalMethodName='saveCollection']"/>                        

                        <!-- Based on the To address of the email, you can create different transformations -->

                        <!-- by using a different mode. -->

                        <xsl:choose>

                            <xsl:when test="$emailContactsNode/emailContact[typeId='1'][emailAddress=$inbox1_emailAddress]">

                                <xsl:apply-templates select="$actionsNode" mode="inbox1"/>

                            </xsl:when>

                            <xsl:when test="$emailContactsNode/emailContact[typeId='1'][emailAddress=$inbox2_emailAddress]">

                                <xsl:apply-templates select="$actionsNode" mode="inbox2"/>

                            </xsl:when>

                        </xsl:choose>

                    </actions>

                </root>

            </actionBatchXml>

        </parameters>

        <xsl:copy-of select="stepPackage/returnXml"/>

    </stepPackage>

</xsl:if>

</xsl:template>
