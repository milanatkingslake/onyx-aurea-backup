Working with GUIDs

Globally Unique Identifiers (GUIDs) are generated by the middle tier when a user saves a new record to a transactional table. GUIDs are unique values that are not duplicated. An example of a GUID is F691E5CE-F97A-49ED-820C-1CBC12F91EF8. GUID values are stored using the UNIQUEIDENTIFIER data type in SQL Server.

The Onyx standard is that all transactional tables have a simple key column containing a GUID value. In contrast, domain tables generally contain a generated domain identifier key (DID) or a natural key (as explained in Primary Key Columns).

GUIDS are not a factor when you are customizing the stored procedures provided by Onyx as source code. However, if your customization requires that you create a stored procedure to replace the SQL generation of an insert method, you must link the middle-tier generated GUID to a parameter for the stored procedure.

Note: For the messages, work_ticket_audit_log, and incident_audit_log objects, the GUID is not generated by the middle tier. Instead, it is allocated by the system prior to calling the stored procedure, by the SQL Server function newid().
