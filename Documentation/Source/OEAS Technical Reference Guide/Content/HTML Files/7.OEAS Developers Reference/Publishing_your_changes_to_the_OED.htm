Publishing your changes to the OED

Caution: The changes that you make to OED metadata can affect the functionality of dependent products such as Onyx Enterprise Studio and Onyx Employee Portal. Before publishing to a production environment, you must carefully verify the changes in a test setting. When you are certain of the ramifications, you can rerun this publish process in the production environment.  

To publish your changes:

  1. Before updating the data, save and validate the OED.
  2. Click to publish your changes to the OED and click OK.

    3. To learn what processes take place when you publish your changes, see Publishing OED customizations.

  3. At the login screen, enter the SQL user ID and password that conforms to the least privilege security for this OEAS installation. The user ID should have the built-in "db_owner" role privileges on the database.

    4. If publishing succeeds, a note appears in the status bar. Otherwise, the Object Designer reports issues and problems in a summary window. View and resolved the errors in the summary. If you ignore them, the Object Designer may automatically mark the object as invalid. Invalid objects are not available to the Onyx Enterprise Application Server (OEAS).

    If no critical problems occurred, the Object Designer copies the XML from the working copy OED to the production copy OED on the same computer. Object Designer also updates the tables and stored procedures by making the necessary schema changes. After the schema changes, the Object designer regenerates and recompiles the OEDB stored procedures.  

Important: You might see a "permission denied" failure at the very end of the publish process. This indicates that the user account the web site is operating under does not have permission to stop COM+. You can stop COM+ manually or use the OD Windows Authentication web site under an administrator account.

    When the OnyxEASApp COM+ component restarts, the OED Cache refreshes and the changes you made are available to users.
