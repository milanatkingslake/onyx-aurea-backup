Relationship Attributes
Relationship attributes

The relationship attributes define a given relationship. In general you use relationships for ensuring data integrity. For transactional objects, SQL generation constructs concurrency control code based on the relationships you specify. The relationship you define between transaction objects is also used to cascade (or restrict) delete actions from parent objects to child objects.

You must know how to construct the metadata to describe the relationship you want. To determine the types of realtionships required by your customization, ask yourself the following questions:

  • Is one or more of the related objects a domain object? (domain)
  • Does this object associate two other objects? (association)
  • Is there an owner or parent - child or master - detail situation involving this object?
  • Are different types of parents allowed? (linkObject)
  • Is it possible for the child to be a collection? (masterDetail)
  • Is this a straightforward relationship between transactional objects? (direct)
  • Is this a relationship between equals? Is the type of object at one end of the relationship allowed to vary? (variableDirect)
  • What do you want to happen when a delete is performed?

Remember that object relationships are set up like foreign keys in a database. When specifying the object properties that define the relationship, you must use the primary key properties for one of the objects. That object is the "primary object" in the relationship and the other object is the "secondary object." The relationship is always defined on the secondary object in Object Designer.
