Customizations

The support for customizations in scripting and the run-time XML depends on the object type being customized.

Domain objects

The script cache uses the get_script_cache_items.sql stored procedure to retrieve domain data. The procedure is written to retrieve data from columns in the known schema when the product ships. Therefore the cache does not recognize any customizations to domain objects other than user defined fields (UDFs). The sproc uses the @sec_temp_used variable to control whether it retrieves UDF data. Onyx sets this variable to zero (0) when it ships the product. In order to support UDFs in scripting, you must set the @sec_temp_used variable equal to one (1) and recompile the sproc. Neither OEP nor OES support display of UDFs in scripting domain objects by default.

New custom properties added to Scripting domain objects are not held in the script cache, nor are they present in the run-time XML passed between OEP and the script engine. The only way for a customization to access the new custom properties is to customize OEP to make additional method calls to retrieve the appropriate customized values. This type of customization might negatively impact performance and be a more complicated customization than normally required in the UI when you add new properties.

Transactional objects

The Scripting feature supports customization of its three transactional objects: scriptSession, scriptSessionStep, and scriptSessionPromptResponse. The script engine persists and retrieves any custom properties and attributes added to these objects. With few exceptions, the script engine does not modify objects in the transactional data section of the run-time XML. Any application sending run-time XML to the script engine must provide the attributes and properties required to call the transactional object methods.

This document does not specify any elements within the transactional object elements. The standard object design for the transactional objects drives which properties are required in a given situation.

Note: The script engine does access properties within the transactional objects for the purposes of error logging or for persisting session state (stateXml).
