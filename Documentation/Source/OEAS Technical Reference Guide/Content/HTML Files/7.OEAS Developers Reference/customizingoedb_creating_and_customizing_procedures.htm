Creating and customizing procedures

OEAS calls procedures through the Onyx Data Manager (ODM) in two different ways. These two ways are named based on the terminology used to describe the way procedures are mapped to the logical model's methods.

  • replace mode - When a replace map is used, OEAS composes a direct procedure call and no T-SQL code is generated.
  • pre or post mode - When a pre or post map is used, the ODM generates an anonymous block of T-SQL code and an in-line procedure call is placed into this block of T-SQL code according to the map type. A pre type of map causes the in-line procedure call to be placed immediately after the declaration section of the T-SQL block, before any other processing instructions. A post type of map causes the in-line procedure call to be placed immediately before the final exception-handling block, after any other processing instructions. The pre- and post- maps are included in the same database transaction as the generated T-SQL code.

Naming procedures

Because the Object Designer user interface organizes user-defined procedures and functions together in the same drop-down list, be sure to name your procedures and functions so that they are easy to recognize while working in the Object Designer.

Because procedures are mapped to specific steps in the Onyx Enterprise Dictionary (OED), you can identify replace, pre, and post procedures by their mapType parameter values of replace, pre, and post, respectively.

Note: For a procedure to be called by the ODM it must be mapped to the specific step that calls the ODM within a logical object's method, as described in Mapping Stored Procedures. The only other way to call a procedure from the middle tier is by writing a custom COM+ object that employs the IStep interface, calling it from a logical method step and having it compose the XML describing the procedure call which it would then pass to the Query Engine (QE). Custom COM+ objects and procedure calls through the QE interface are not described in this document.
