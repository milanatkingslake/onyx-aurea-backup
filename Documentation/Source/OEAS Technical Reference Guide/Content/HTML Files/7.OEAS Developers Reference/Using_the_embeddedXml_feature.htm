Using the embeddedXml feature

Using the embeddedXml feature

OEAS retrieveList and retrieveCollection methods use the embeddedXml feature by default. If you write a custom step component that retrieves a lot of data from the database in a single call, particularly when the data is returned in numerous XML nodes and child nodes, you might want to use the embeddedXml feature.

Onyx created the embeddedXml feature to enhance the performance of methods that retrieve large lists of data. This element uses the XML construct called CDATA so that the returned data is handled as a string of data that is excluded from all parsing operations. OEAS also supports this feature anywhere in the step package, including within the customData element.

The embeddedXml feature consists of two literal strings. The Onyx Transaction Manager (OTM) searches for these strings in the step package that is returned by the last step in an OTM call. If it finds the strings, it removes them from the stepPackage XML. The literal strings are:

<embeddedXml><![CDATA[

and

]]></embeddedXml>

Do not use the embeddedXml feature to pass data that is needed by a subsequent step in the same OTM call. For example, if you include in a CDATA section an access code that a subsequent step component needs, the step component will not find it, because the parser cannot find anything within the CDATA section.

Because the Internal OTM (IInternalTransactionManager interface) does not parse XML, it does not support the embeddedXml feature. To call a retrieveList or retrieveCollection method, you should do one of the following:

  • Disable the embeddedXml feature for the method you are calling.
  • Call the method using the ITransactionManager interface of OTM instead, which supports the embeddedXml feature.
Disabling the embeddedXml feature

As stated previously, retrieveList and retrieveCollection methods use the embeddedXml feature by default, however, OEAS disables the embeddedXml feature for certain methods such as retrieveCollection on the emailAttachment object. For your custom(ized) retrieveList or retrieveCollection methods, you must disable embeddedXml in the following cases:

  • The method has a step following the Onyx Data Manager (ODM) step that needs to inspect or change the values in the returned collection or rowSet.
  • The method is called by some step component via the internal OTM.

The emailAttachment.retrieveCollection and literatureOrder.retrieveCollectionBatch methods are examples of retrieveCollection methods called by the internal OTM. In the case of emailAttachment.retrieveCollection, the OnyxEmailSendStep calls emailMessage.retrieve, which subsequently calls emailAttachment.retrieveCollection. In this case, embeddedXml must be disabled so that the email send step can access the returned emailAttachments collection. If you look at the core step for the emailAttachment retrieveCollection method in Object Designer, you see that the Additional Data attribute includes the following name-value pair:

disableEmbeddedXml=1;

Apply note style for this:

You disable embeddedXml on the retrieveList or retrieveCollection method being called by the internal OTM not on the method calling the internal OTM.

You can disable the embeddedXml feature by adding this name-value pair to the Additional Data attribute on any retrieveList or retrieveCollection step that calls ODM, as follows:

stepName=core;

stepObject=myCustomObject;

stepMethod=retrieveCollection;

dataPath=mySpecialObjects;

disableEmbeddedXml=1;

How embeddedXml works

How the embeddedXml element works:

  1. An OEAS retrieveList method calls a step component that typically returns a large list of data from the database.
  2. The step component builds the rowSet data as an XML string, and inserts the data within the embeddedXml element.

The embeddedXml element begins with exactly:

<embeddedXml><![CDATA[

and ends with exactly:

]]></embeddedXml>

    Note that the code must appear exactly as illustrated above: it cannot contain any white space between the embeddedXml and CDATA elements, not even the hard return that is sometimes used when formatting XML for visual display.
