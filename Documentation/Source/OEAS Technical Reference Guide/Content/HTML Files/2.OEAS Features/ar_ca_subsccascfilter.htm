Cascade only the secondary objects that meet this filter expression

The Cascade only the secondary objects that meet this filter expression rule can be used to limit cascaded subscriptions to a defined subset of an object type (for instance a particular incident or task category). It can be set at the relationship level and/or the role level, but can only be set if you have enabled the Subscribe secondary object and cascade its subscription relationships rule.

Providing a filter expression for a subscription relationship ensures that when an end-user subscribes to the primary object in the relationship, an automatic (cascaded) subscription to the relationship’s secondary objects is created for that subscriber, but only to those secondary objects that meet a specified filter expression criteria.

In addition, the cascade process also checks for rules on any relationships of the secondary objects and, if found, will subscribe those objects’ related objects as well, both primary and secondary, where directed by those rules. The cascade process continues to auto-subscribe records to the subscriber in this manner until there is no further relationship to process. For example, when the top-level object is reached, when a duplicate (already subscribed) object is reached, or when an object is reached that does not have a subscription relationship with cascade rules.

The original subscription to the primary object can be a result of the end- user explicitly subscribing to the object herself, an advanced subscriber subscribing the object to the user, or another cascade process. In any case, once the subscription is created, the cascade process will proceed as described when this rule is set to true.

Example:

Suppose you have the following:

  • A subscribable company object.
  • A subscribable incident object with a subscription relationship between the company and incident objects based on the OEDB company owner of an incident relationship, for which the Cascade only the secondary objects that meet this filter expression rule is set to “iIncidentCategory=3” so that cascaded subscriptions are created only for sales incidents.
  • A subscribable task object with a subscription relationship between the incident and task objects based on the OEDB incident owner of a task relationship, for which the Subscribe secondary object and cascade its subscription relationships rule is set to true.

In this case, when an end-user subscribes to a company, cascaded subscriptions are created for the end-user to all of the “sales” type incidents owned by that company, as well as to all of the tasks owned by those sales incidents.
