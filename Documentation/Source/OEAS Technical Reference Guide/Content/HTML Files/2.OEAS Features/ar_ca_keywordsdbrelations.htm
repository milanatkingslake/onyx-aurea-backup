Database table relationships

The underlying database structure that supports the keyWord business object is illustrated in the following graphic.

Tags are persisted to the database by the incident, task, workTicket, company, individual, and customerProduct objects, and details regarding the tags are persisted to both the key_word_association and key_word tables.

  • The key_word table contains the actual word or phrase entered as the tag and the object identifier (object_type_enum) specifying whether the tag is associated with an incident, task, workTicket, company, individual, or customerProduct object. In the case of an incident or task object, the key_word table additionally identifies the incident or task category (category_did). This table contains a single instance of a tag for each object type/category, no matter how many times that tag is used. Additionally, this table stores the number of times that a particular tag has been used for that object type/category (in the use_count field). When an instance of a tag for a particular object type/category reaches a certain number (as specified in the TagIncludeCount system parameters), that tag becomes a tag lookup value (for that particular object type/category).
  • The key_word_association table stores the information that links each tag (stored in the key_word table) to the particular incident, task, or workTicket object to which it is associated. Based on this relationship, a particular tag can be associated to many incidents, tasks, or work tickets. This also enables you to search the OEDB and retrieve all incidents, tasks, or work tickets associated with a particular tag.

Tags data is not case sensitive. The tags “procedure” and “Procedure” are saved as a single instance of a tag.
