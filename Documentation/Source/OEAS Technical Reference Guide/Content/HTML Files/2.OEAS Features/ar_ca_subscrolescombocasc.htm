Combining role-specific rules during cascade

There are several scenarios to consider when you have created role-based rules for subscription relationships:

  • An OEP end-user can be a member of an organizational role that is associated with more than one subscription relationship.
  • An OEP end-user can be a member of more than one organizational role and, as such, can be included in more than one subscription role.
  • The rule set for a subscription role overrides the same rule set for the subscription relationship for members of that role.
  • An OEP end-user who is associated with multiple subscription roles for the same subscription relationship is subscribed to a union of all of the subscribable records based on those roles.

The following example demonstrates how these scenarios are combined for cascaded subscriptions.

Example

Suppose you have four OEP end-users who belong to the following organizational roles, and for whom you have created subscription profiles for OEP-Offline.

Suppose that you also have the following subscribable objects and the following subscription rules set for those subscribable objects:

  • individual
  • incident
  • task
  • forecast

Note that for this example:

  • There is only a requirement that there be an individual subscribable object. There are no subscription relationships defined for individual, as the only relationships required for this example exist on the secondary object (incident).
  • Subscription roles exist only for the incident object’s individual owner of an incident relationship and the task object’s incident owner of a task relationship.

Based on the subscribable objects, relationships, rules, and roles described above, the following end-user interactions result in the cascaded subscriptions described:

User 1 subscribes to an individual

  • User 1 belongs to the A organizational role, which is not associated with a subscription role.
  • Relationship rules are used to determine cascaded subscriptions. No role-based rules are used.
  • The Cascade secondary object and cascade its subscription relationships is set to true for all subscription relationships, so subscriptions are cascaded recursively for the incidents owned by the subscribed individual, and the tasks and forecasts owned by those subscribed incidents.

This scenario would be the same for any subscriber who was not a member of an organizational role.

User 2 subscribes to an individual

  • User 2 belongs to the B organizational role, which is associated with a subscription role on the incident individual owner of an incident relationship.
  • The rules on this subscription role are set to false.
  • The false setting for Subscribe to secondary object and cascade its subscription relationships overrides the true setting for this rule that is set at the relationship level. Because of this, no secondary incidents are subscribed to User 2 during the cascade.

User 3 subscribes to an individual

  • User 3 belongs to the B and C organizational roles, which are each associated with a subscription role on the incident individual owner of an incident relationship.
  • The rules on the subscription role associated with role B are set to false, and the false setting for Subscribe to secondary object and cascade its subscription relationships overrides the true setting for this rule that is set at the relationship level.
  • On the subscription role associated with role C, Subscribe to secondary object and cascade its subscription relationships is set to true, with an additional filter criteria set to iIncidentCategory=1.
  • Based on a union of these rules, subscriptions for User 3 are cascaded recursively to the “support” type incidents owned by the subscribed individual, and the tasks and forecasts related to those incidents.

User 4 subscribes to an individual

  • User 4 belongs to the B, C, and D organizational roles, which are each associated with a subscription role on the incident individual owner of an incident relationship.
  • The rules on the subscription role associated with role B are set to false, and the false setting for Subscribe to secondary object and cascade its subscription relationships overrides the true setting for this rule that is set at the relationship level.
  • On the subscription role associated with role C, Subscribe to secondary object and cascade its subscription relationships is set to true, with an additional filter criteria set to iIncidentCategory=1.
  • On the subscription role associated with role D, Subscribe to secondary object and cascade its subscription relationships is set to true, with an additional filter criteria set to iIncidentCategory=3.
  • Based on a union of these rules, subscriptions for User 4 are cascaded recursively to the “support” and “sales” type incidents owned by the subscribed individual, and the tasks and forecasts owned by the subscribed sales and support incidents.
