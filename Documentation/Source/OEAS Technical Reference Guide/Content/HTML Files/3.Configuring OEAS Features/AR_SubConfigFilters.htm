Configuring filters for Subscription Designer’s filter helper menu

You can set the Cascade only the secondary objects that meet this filter expression rule for a subscription relationship at both the relationship level and the role level, as described in Cascade only the secondary objects that meet this filter expression. You set this rule in Subscription Designer, where you can either manually type the filter expression into the provided text field, or use the filter helper menu to select an appropriate filter expression.

With installation of OEP-Offline, the filter helper menu is only active for incident and task relationships, and the only filter expressions available are for the incident and task categories of sales, service, and support. However, the behavior and contents of the filter helper menu is controlled by the customizations.xml configuration file, which you can configure to support additional relationships and filter expressions. By updating the configuration file you can add filter expressions to the filter helper menu that are available for all subscribable objects and their relationships or only for specific subscribable objects and their relationships.

The following table describes the metadata for customizations.xml.

element (*) indicates a required element Value Description

<filters>(*)

1

The <filters> collection can contain any number of <filter> elements.

<filter>(*)

0..*

Each <filter> element must include the

following sub-elements:

  • <description>
  • <expression>

<description>(*)

1

The <description> element contains the text that you want the filter helper menu to display to the user.

<expression>(*)

1

The <expression> element contains the filter expression that is assigned to the rule when a user selects this filter from the filter helper menu.

The expression must be in the following format:

child.<field><SQL operator><value> where child represents the secondary object in the relationship. OR parent.<field><SQL operator><value>, where parent represents the primary object in the relationship.

<supportedObjectTypes>(*)

1

This collection can contain any number of <primaryObjectType>,

 <secondaryObjectType>, and <eitherObjectType> elements. When

 a user clicks the filter helper menu button, the filter is included in the menu if both of the following conditions are true:

  • The <supportedObjectTypes> collection contains a <primaryObjectType> or <eitherObjectType> element that has a value of any, or a value that matches the primary object type for the relationship or role being edited.
  • The <supportedObjectTypes> collection contains a <secondaryObjectType> or <eitherObjectType> element that has a value of any, or a value that matches the secondary object type for the relationship or role being edited.

A filter must always contain at least one <primaryObjectType> and at least one <secondaryObjectType>, or at least one <eitherObjectType>, or it will never meet both of these conditions and thus will never be displayed in the filter helper menu. If you want a filter to always be displayed in the menu, use <eitherObjectType>any</eitherObjectType>.

<primaryObjectType>

 

This element contains the type code of an object

 (1=individual, 2=company, etc.), or any.

<secondaryObjectType>

 

This element contains the type code of an object

 (1=individual, 2=company, etc.), or any.

<eitherObjectType>

 

This element contains the type code of an object

 (1=individual, 2=company, etc.), or any.

Note: There is no validation performed on filter expressions. If you provide an expression that is not valid SQL, or that references a field that does not exist in the secondary object’s underlying database table, the filter expression could be disregarded, or cascading subscriptions for that relationship could fail altogether.

Example 1

This filter expression appears in the filter helper menu for any relationship where an incident (object type 6) is the secondary object in the relationship. The filter is for cascaded subscriptions only to those incidents where incident_category_did=1 (support incidents).

<filter>

<description>Support Incidents Only</description>

<expression>child.incident_category_did=1</expression>

<supportedObjectTypes>

<primaryObjectType>any</primaryObjectType>

<secondaryObjectType>6</secondaryObjectType>

</supportedObjectTypes>

</filter>

You can modify this filter by replacing <primaryObjectType>any</primaryObjectType> with <primaryObjectType>1</primaryObjectType>, in which case the filter expression appears in the filter helper menu only for those relationships where individual (object type 1) is the primary object and incident is the secondary object.

Example 2

This filter expression appears in the filter helper menu for all relationships because the primary object type can be any object, and the secondary object type can be any object. The filter is for cascading subscriptions only to those secondary objects that have been updated subsequent to 12/31/02.

<filter>

<description>Records updated on or after 12/31/02</description>

<expression>child.update_date > ‘12/31/02’</expression>

<supportedObjectTypes>

<eitherObjectType>any</eitherObjectType>

</supportedObjectTypes>

</filter>

You can modify this filter by replacing <eitherObjectType>any</eitherObjectType> with <eitherObjectType>6</eitherObjectType>, in which case the filter expression appears in the filter helper menu only for those relationships where incident is either the primary object or the secondary object.
