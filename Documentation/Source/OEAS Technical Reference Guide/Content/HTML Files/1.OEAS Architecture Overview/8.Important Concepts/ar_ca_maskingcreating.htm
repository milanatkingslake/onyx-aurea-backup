Creating Masks

You can modify/create a mask for the following business objects:

  • referenceField (mask property)
  • country (postCodeMask property)
  • country (phoneMask property)

You modify and create masks for these objects using the Table Administration and Reference Table Administration tools in the Onyx Enterprise Studio, modifying and creating masks for phone numbers, postal codes, and user-defined configurable fields.

To modify the default masks for phone numbers and postal codes, use Table Administration (selecting Country from the Table list in the navigation area). For additional information, see Customers and Table Administration. For additional information on creating user-defined fields, see Reference Table Administration Help.

The following table identifies the characters that are reserved for formatting, and what each of these characters represents. All other characters or symbols are treated as literals and appear in the masked text as shown to the user.

Leading spaces in a data field are trimmed by OEAS when the contents of the field are written to the database. If you create a mask that allows spaces as the first user-enterable items in a masked data field, they might not be saved when the data is saved, causing unexpected behavior when the data is later retrieved and displayed to tje user.

Supported character Masked values

9

Accepts any numeric digit '0' - '9', or a decimal point

A

Accepts alphabetic characters 'A' - 'Z' and 'a' - 'z'

D

Accepts any numeric digit '0' - '9', or a space  

H

Accepts any hexadecimal value: '0' - '9', 'A' - 'F', and 'a' - 'f'

L

Accepts alphabetic characters: 'A' - 'Z' and 'a' - 'z'. If an uppercase character is entered, it is converted to lowercase.

N

Accepts alphanumeric characters: '0' - '9', 'A' - 'Z', and 'a' - 'z'

U

Accepts alphabetic characters: 'A' - 'Z' and 'a' - 'z'. If a lowercase character is entered, it is converted to uppercase.

X

Accepts any valid Unicode character.  

/

Converts a mask character to a literal. To display a forward-slash character as a literal, use two consecutive forward-slash characters. For example, “D//D” displays a digit entry position followed by a single forward-slash, followed by another digit entry position.

Do not use the literal identifier (forward-slash) in front of another literal (non-mask) character. Unexpected results might occur. The literal identifier should only be paired with one of the masking characters that appear in this table.

. , : ; -

Characters used as separators for date, time, and numerical values, depending on your international settings

To understand how these reserved characters can be used to create a mask, consider the following example for a simple phone mask:

  • 999 999-9999/x 99999

which enables users to format phone numbers as:

  • 487 555-1212x 56789

The escape character "/" is used to display "x" as a literal character, here to represent an extension to the phone number. The "9" permits any numerical character 0-9.

Note that additional masking is usually performed within the presentation tier to facilitate data entry (so that end users can enter only numbers and no spaces or special characters).

Supported character Masked values

As stated previously, you define this mask for the country object using the Table Administration tool. OEAS and Onyx Employee Portal (OEP) share the same masking data in the country table. OEP supports additional postal code masking characters. For more information on which characters OEP supports, see the OEP documentation. If you choose to use the additional characters supported by OEP, do not set the customer retrieveList, retrieveListByWorkTicket, or retrieveSummaryInfo includeMask parameter to ‘apply’, or you might get unexpected results in the postal code field.
