Understanding Fulfillment Subsystems

Fulfillment subsystems process order lines. Each fulfillment subsystem is configured either to print or to send an email. A fulfillment subsystem configured to print is a print subsystem. A fulfillment subsystem configured to send email is an email subsystem. You can have multiple fulfillment subsystems, each with unique settings.

Order lines are processed according to the attributes of literature products, order lines, and subsystem settings.

Fulfillment subsystems retrieve order lines for processing based on the following:

  • Subsystem operation window
  • Subsystem run interval
  • Subsystem batch size
  • Subsystem active status
  • Order line status
  • Order line priority
  • Order line fulfillment type
  • Order line insert date
  • Order line scheduled process date

The following sections discuss these settings.

Subsystem operation window. A subsystem's operation window is the time frame within which the subsystem processes literature orders. You can configure a subsystem to process 24 hours a day or just within a specified time period defined by the Start Time and Stop Time settings. To configure a subsystem to process 24 hours a day, set both the Start Time and Stop Time to 00:00.

Subsystem run interval. The run interval is the time between subsystem checks of the new order lines queue for order lines with a status of new. Fulfillment subsystems have two intervals: standard priority and high priority. During the standard-priority interval, both high and standard-priority orders are processed. This interval is defined by the Standard Priority Run Interval setting. During the high-priority interval, only high-priority order lines are processed. This interval is defined by the High Priority Run Interval setting.

Subsystem batch size . The batch size is the approximate number of order lines that a subsystem retrieves for processing at one time. Several rules affect the actual number of order lines included in a batch:

  • Orders are never split. If the batch size is reached in the middle of an order, the remaining order lines are included in the batch.
  • Order lines for mailing labels are counted as part of the batch size.
  • Order lines for system documents are not counted as part of the batch size.

Suppose that you have a batch size set to 1000 and the New Order Lines queue has 2000 orders, each with two order lines. When the subsystem retrieves the order lines, it picks up only 500 orders for processing, because batch size is the total number of order lines (500 orders x 2 order lines each = 1000 order lines).

Subsystem active status. You can configure a subsystem as active or inactive using OPS Administration. When the OPS service is started, active subsystems also start while inactive subsystems do not.

However, you can explicitly start an inactive subsystem. When the OPS service is stopped or paused, all subsystems with a status of started, regardless of active status, also stop or pause accordingly.

Order line status. The order line status is for tracking order line processing. A subsystem retrieves only order lines that have the status of New. When a new order line is retrieved, its status is updated to In Process. When processing is complete and the order line has been printed or emailed successfully, the status is updated to Completed. In the event of a failure, the status is updated to Error.

Order line priority. The order line priority indicates either immediate or standard handling. A subsystem retrieves immediate priority items within the next high priority interval, otherwise it retrieves standard priority items during standard priority intervals according to the other queuing criteria.

Order line processing dates. Order lines are processed first-in-first-out, which means that order lines with the oldest insert date are processed first. The exception to this rule are order lines with a scheduled process date set to a date in the future. These order lines are not processed before this date.
