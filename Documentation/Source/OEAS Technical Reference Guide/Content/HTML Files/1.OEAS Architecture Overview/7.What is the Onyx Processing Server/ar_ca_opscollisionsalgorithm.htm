Collision-Resolution Algorithms

The Collision Handler attempts to automatically resolve collisions between versions of the same record when:

  • The offline user and the online user both make the same change, such as changing the status of an incident to the same value.
  • The offline user or the online user saves a record without making any changes. This situation happens when either the offline or online user adds work notes to an incident but does not change the incident header information.
  • The differences between versions are limited to values in maintenance and read-only fields, which can be configured for each business object type.

The record that is automatically preserved depends on uses one of three algorithm settings:

  • Online wins—the record on the OEDB or staging server is preserved.
  • Last update wins—the record with the most recent update date is preserved.
  • Customer—special collision processing for Individuals and Companies.

Customer is a special algorithm for Companies and Individuals. When differences between versions are limited to maintenance and read-only fields of the primary record (Company or Individual) and the secondary records (Phones and Addresses), the Collision Handler can automatically resolve collisions. When this resolution is successful, the most recently updated version of the record is preserved.

When the Collision Handler cannot resolve a collision automatically, it creates a conflict that the user must help to resolve.

The default algorithm settings for the record types that users can subscribe to out of the box are listed in the following table.

Record type

Algorithm

Comments

Individual

customer

Various read-only and maintenance fields in the Individual, Phone, and Address records are ignored and will not cause collisions.

Company

customer

Various read-only and maintenance fields in the Company, Phone, and Address records are ignored and will not cause collisions.

Incident

lastUpdateWins

Various read-only and maintenance fields in the Incident header are ignored and will not cause collisions.

Task

lastUpdateWins

Various read-only and maintenance fields in the Task header are ignored and will not cause collisions.

Forecast

N/A

The Collision Handler is not used. The sync subsystem creates a conflict when Forecast header and lines are out-of-date.

Quote

N/A

The Collision Handler is not used. The sync subsystem creates a conflict when Quote header and lines are out-of-date.

Work note

N/A

The Collision Handler is not used. Work notes cannot be updated or deleted; they can only be inserted.

For more information about the Collision Handler settings, see the OEAS Developer’s Reference and the Help for the Subscription Designer, which is an Onyx Enterprise Studio (OES) Administration tool.
