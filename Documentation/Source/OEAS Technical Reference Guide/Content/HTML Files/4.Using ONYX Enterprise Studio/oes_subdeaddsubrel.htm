Adding a subscription relationship
Adding a subscription relationship
  1. In the navigation area, navigate to and expand the appropriate subscribable object to which you want to add a subscription relationship, and select the Relationships node.

The Relationships view appears in the edit area.

  1. In the edit area, select the relationship that you want to add as a subscription relationship. The list of available relationships contains only those relationships that exist in the OED that have not yet been created as a subscription relationship.

Even though company_incident_contactId and company_task_contactId are listed, they are not used in OEP. Adding these relationships to incident or task objects has no effect unless you have customized OEP to use them.

  1. Click Make Subscription Relationship.

A new subscription relationship (with the same name as the OED relationship you chose) appears in the navigation area followed by an asterisk (*).

  1. In the edit area, define the attributes for the subscription relationship.

ID - The ID attribute is not editable as it is a primary key field. This attribute is a unique application-defined identification number given to each subscription relationship created.

OED Relationship - This attribute is not editable. It identifies the underlying OED relationship that you chose for the subscription relationship.

Relationship Description - This attribute is not editable. It is the OED relationship description for the relationship that you chose for the subscription relationship.

Name - This attribute is the user-friendly name you provide for the subscription relationship. There is no requirement for the name to have any particular relation to the underlying OED relationship. The name can be a maximum of 50 characters.

Description - The Description attribute is the brief description of the subscription relationship that you provide. The description can be a maximum of 255 characters.

Active - The Active attribute controls whether or not the subscription relationship is recognized by the cascade process so that any rules or roles associated with it can be used to cascade related objects.

Do not set a subscription relationship to inactive after deployment of OEP-Offline. Once the relationship is inactive, unsubscriptions and deletions are not cascaded to the secondary objects of the relationship, which can result in orphaned records in OEP-Offline. In the case of deletions, the subscription_item_cleanup.sql script (see Removing deleted data from subscription_item) removes the orphaned records. In the case of unsubscriptions, you must remove the orphaned records manually.

  1. Define the rules for the subscription relationship. These rules define automatic (cascading) subscription, unsubscription, and deletion behavior between the primary and secondary objects in a relationship. For more information about the impact of these rules, see Subscription relationships, rules, and roles.

Always subscribe the primary object - Selecting this attribute for the subscription relationship ensures that when a subscriber subscribes to the secondary object in the relationship, an automatic cascaded subscription is created for that subscriber to the primary object in the relationship.

The Always subscribe the primary object rules that are provided by default are required rules. Changing these default rules to false can adversely affect the display of records in OEP-Offline. It can also cause incomplete unsubscription and deletion cascades resulting in unnecessary synchronization of records and orphaned offline records (records that the end-user cannot access).

Subscribe primary object and cascade its subscription relationships - Selecting this attribute for the subscription relationship ensures that when a subscriber subscribes to the secondary object in the relationship, automatic cascaded subscriptions are created for that subscriber to the primary object in the relationship.

Subscribe secondary object and cascade its subscription relationships - Selecting this attribute for the subscription relationship ensures that when a subscriber subscribes to the primary object in the relationship, automatic cascaded subscriptions are created for that subscriber to the secondary objects in the relationship.

Cascade only the secondary objects that meet this filter expression - This attribute contains a SQL expression used in a WHERE clause that limits the automatically subscribed secondary object instances. Providing a filter expression for the subscription relationship ensures that when an end-user subscribes to the primary object in the relationship, automatic cascaded subscriptions are created for that subscriber to only the secondary objects in the relationship that meet the filter expression. This rule can only be set if you have enabled the Subscribe secondary object and cascade its subscription relationships rule. This attribute can be a maximum of 500 characters.

The filter expression must be valid SQL. Invalid SQL can cause cascades involving this relationship to fail. For information about creating a valid filter expression, see Filtering secondary object cascaded subscriptions. .

  1. Click to save your changes.
