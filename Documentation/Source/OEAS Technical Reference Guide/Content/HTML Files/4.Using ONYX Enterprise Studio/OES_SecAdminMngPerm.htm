Managing permissions for resources

Managing permissions from within the Resources node in Security Administration involves the following:

  • Business Objects - In the Business Objects node, you can set access permissions for an object’s methods and for specific actions, such as creating a new incident (incident.insert).
  • UI Resources - In the UI Resources node, you can manage permissions related to a user interface (OEP, OES tools, for example). UI resource’s can represent either an entire user interface, such as it does for the OES tools, or they can represent individual areas of a particular user interface, allowing for a more granular level of security.
  • Other resources- In the Other Resources node, you can grant access permissions for your customizations. See Managing permissions for resources for further details.

In addition, you can manage permissions for a Business Object resource that a user interface uses. When the actions of the user interface are performed by objects and methods defined within the OED, such as with the OES tools and OEP, you can manage the permissions related to these actions.

For example, if a user or role requires read-only access to the Contact Type Administration tool, you would:

  • Grant the user or role access to Contact Type Administration in UI Resources. See Granting a user or role access to a resource.
  • Secure the Contact Type object’s saveCollection method as a resource in Business Objects. See Securing business objects methods for instructions. Once secured, all users and roles will have read-only access permissions to Contact Type Administration.
  • Since the Contact Type object’s saveCollection method is now secured as a resource, you should grant permission to the users or roles that need the ability to insert, update, or delete contact types in Contact Type Administration.
  • Add a user or role to a resource
  • Grant a user or role access to a resource
  • Deny a user access to a resource
  • Remove a user or role from a resource
